详细分析：技能提取阶段的已有技能摘要注入与 Token 优化
一、注入时机与范围
当前现实：提取阶段完全没有注入已有技能摘要
让我们追踪 skill_extraction 场景的完整调用链：
tick_loop (每 interval_ms 触发)
  └─ tick_inner()
      └─ run_scenario_loop()           [service.rs:498]

          │
          ├─ evaluate_all()            [service.rs:530]

          │

          │   ├─ format_recall_for_prompt() → "Previously Learned Context"
          │   └─ ⚠️ 这是**通用召回**，不是针对去重的技能指纹列表
          │
          └─ call_llm_for_scenario()   [service.rs:853-901]
              └─ max_tokens: 4096, temperature: 0.7, thinking_enabled: false
关键发现：
ScenarioContext（types.rs:44-55）没有 existing_skills 字段——这是根本性的结构缺位。
第549-577行的 MemoryRecallEngine 注入确实提供了"Previously Learned Context"，但它是查询驱动的通用召回（用场景名称+描述作为 query），不是去重专用的技能指纹列表。它返回的是完整 SOP 文本（context + principles + steps + pitfalls），对去重来说过于臃肿。
skill_extraction.rs 的 build_context() 完全不知道已有技能的存在。系统 prompt 第37行说"不是已有 skill 的重复"，但 LLM 处于"盲人摸象"状态。
正确注入位置
已有技能摘要应在 两个位置 注入，各司其职：
位置	注入内容	目的
位置A：skill_extraction.rs::build_context() 的 context_messages	紧凑的技能指纹列表（title + description，≤80字符/条）	让 LLM 在生成前逐条对比，避免输出重复技能
位置B：service.rs 第549行的 MemoryRecallEngine 增强	已有技能完整 SOP（当前已有，保持不变）	让 LLM 理解已有技能内容以便做语义判断
位置A是新增的，位置B是现有机制但需要调整排序策略。
注入的数据范围
不应注入所有技能，应按以下策略分层：
第一层（必有，≤50条）：当前 workspace 下 enabled=true 的 promoted learned skills
  排序：cited_count DESC → usage_count DESC
  字段：name + description（≤80字符/条）

第二层（按需，≤20条）：当前 workspace 下 enabled=true 的 draft learned skills  
  排序：updated_at DESC
  字段：name + description
  
总计上限：70条 × 80字符 ≈ 5,600 字符 ≈ 1,400 tokens（含格式开销约 2,000 tokens）
为什么不注入所有技能：
draft 技能未经使用验证，大部分是噪音
如果用户有 500 条学得技能，全量注入会消耗 40,000+ tokens，远超合理预算
cited_count 排序确保最活跃/最可靠的技能在去重时优先级最高
二、Token 消耗控制机制
当前机制的 Token 消耗分析
让我们逐段估算 skill_extraction 场景一次调用的 token 消耗：
组成部分	内容	估算 tokens
System prompt (基础)	SKILL_EXTRACTION_SYSTEM_PROMPT	~1,600
System prompt (召回增强)	MemoryRecallEngine 返回的技能 SOP	1,000-8,000（变数最大）
执行日志	50条日志，含 tool_name/input/output	~3,000-5,000
最近对话	5条消息	~500-1,500
LLM 输出	max_tokens=4096	最多 4,096
总计		~6,100 - 18,100
最大的 token 消耗来源是召回增强的 "Previously Learned Context"——format_recall_for_prompt 将已有技能的完整 SOP（context + principles + steps + pitfalls）全部写入系统提示。每个技能的 SOP 平均 300-800 字符，3-8 条技能就是 900-6,400 字符。
当前已有的 Token 控制措施
执行日志截断：tool_input 和 tool_output 被序列化为 {"summary": ...}，做了第一层压缩（service.rs:346-347）
对话消息截断：只取最近 5 条（skill_extraction.rs:274）
执行日志数量限制：recent(50)（service.rs:504）
上下文滑动窗口：CONTEXT_WINDOW_SIZE = 20（service.rs:77）
max_tokens 限制：LLM 调用使用 max_tokens: 4096（service.rs:881）
thinking_enabled: false：避免 DeepSeek 类模型的 reasoning_content 消耗（service.rs:884）
缺失的 Token 控制措施
问题1：召回增强注入的是完整 SOP，对去重来说信息密度极低format_recall_for_prompt 为每条技能输出：
### 技能名称
**适用场景**: ...
**核心原则**: ...
**SOP 步骤**: 1. ... 2. ... 3. ...
**注意事项**: ...
这对去重判断来说，80% 的内容是冗余的——去重只需要知道"这个技能是做什么的"（title + description），不需要知道具体步骤。问题2：没有技能摘要的长度限制如果有 8 条技能被召回，每条 500 字符 = 4,000 字符 ≈ 1,000 tokens。如果这些技能的大部分与当前执行日志无关，这些 token 就完全浪费了。问题3：没有基于场景的分级注入策略skill_extraction 场景需要的去重信息与其他场景（如 conversation_learning）不同，但当前使用相同的召回策略。
建议的 Token 优化措施
措施1：为 skill_extraction 场景创建专用的轻量级技能指纹格式
// 在 skill_extraction.rs 或 service.rs 中新增
fn format_skill_fingerprints_for_dedup(skills: &[SkillFingerprint]) -> String {
    let mut out = String::from("## 已有技能参考（请避免生成重复技能）\n\n");
    out.push_str("格式：序号. 名称 | 描述 | 类别 | 引用次数\n\n");
    for (i, s) in skills.iter().enumerate() {
        // 每条 ≤100 字符（含序号和分隔符）
        let desc = truncate_to_char_count(&s.description, 60);
        let cat = s.category.as_deref().unwrap_or("-");
        out.push_str(&format!(
            "{}. {} | {} | {} | cited:{}\n",
            i + 1, s.title, desc, cat, s.cited_count
        ));
    }
    out
}

struct SkillFingerprint {
    title: String,
    description: String,
    category: Option<String>,
    cited_count: u64,
}
每条指纹约 80-100 字符，50 条 = 4,000-5,000 字符 ≈ 1,000-1,250 tokens——这与当前一条完整 SOP 的 token 量相当！措施2：系统提示中移除冗余的完整 SOP，改由 skill_extraction 专用的紧凑摘要替代当前 service.rs:549-577 注入的 "Previously Learned Context" 对去重场景应替换为上述紧凑指纹。完整 SOP 保留给 agent 实际执行时使用，但不应该在提取阶段携带。措施3：执行日志的进一步压缩当前每条日志包含完整的 tool_input 和 tool_output JSON。对于去重判断，失败日志的关键信息是错误类型而非完整输入输出：

    // 基线 2048 + 每条日志 30 tokens + 每条已有技能 10 tokens

三、上下文管理与平衡
当前触发频次控制
skill_extraction 的触发由 SkillExtractionConfig 控制：
参数	默认值	含义
trigger_execution_count	10	每 10 次工具执行触发一次
trigger_on_failure	true	失败时立即触发（冷却减半）
min_interval_ms	120,000	两次触发之间至少间隔 2 分钟
实际触发频次（假设活跃使用）：
正常情况：每 10 次工具执行 + 2 分钟冷却 → 约每 5-10 分钟一次
失败触发：每次失败 + 1 分钟冷却 → 高频场景可能每 1-2 分钟一次
Token 消耗与触发频次的关系：如果每次 skill_extraction 调用消耗 10,000 tokens（输入+输出），每天活跃 8 小时：
正常触发：~48 次/天 → 480,000 tokens/天
高频失败：~120 次/天 → 1,200,000 tokens/天
这还没有算上已有的 recall 注入的 token。如果全量注入 50 条技能指纹（~1,250 tokens），增量可忽略不计。
最优技能数量的确定
去重效果与注入技能数量的关系遵循边际递减规律：
去重效果
  ↑
  │    ┌──────────────
  │   ╱
  │  ╱
  │ ╱
  │╱
  └──────────────────→ 注入技能数
      ↑
   最优区间: 30-50 条
< 10 条：覆盖不足，LLM 无法判断新技能是否与已有技能重复
30-50 条：覆盖了 cited_count 最高的技能（占实际使用量的 80-90%），边际收益拐点
> 100 条：token 消耗线性增长，但去重收益增长趋近于零（长尾技能很少被提取到重复）
建议的动态策略：
fn optimal_skill_count_for_dedup(total_skills: usize) -> usize {
    match total_skills {
        0..=20 => total_skills,       // 少的时候全量
        21..=100 => 30,                // 中等规模取 top-30
        101..=500 => 50,               // 大规模取 top-50
        _ => 50.min(total_skills / 10), // 超大规模取 10%
    }
}
动态调整上下文规模的机制
当前不存在动态调整机制。建议实现以下分层策略：
Layer 0 (始终注入，0 tokens): 系统 prompt 中的去重指令
  "以下是你已学得的技能清单。逐条对比——如果本次日志与其中某条描述的是
   同一个失败模式或同一个解决方案，不要输出新 <skill>。"

Layer 1 (按需注入，500-1,500 tokens): 高质量技能指纹
  - promoted 技能 top-30（按 cited_count DESC）
  - 每条: title + description，≤80字符

Layer 2 (条件注入，额外 500 tokens): 最近技能指纹
  - 最近 7 天内更新的 draft 技能 top-10
  - 仅在最近有新提取时注入（避免总是注入空列表）

Layer 3 (已有机制，0-3,000 tokens): 召回 SOP（保持不变）
  - MemoryRecallEngine 的 "Previously Learned Context"
  - 但应降低在 skill_extraction 场景的 recall limit
该分层策略可将 skill_extraction 场景的 token 消耗从当前的平均 ~10,000 降至 ~6,000-7,000，同时提升去重准确性（因为 LLM 获得了更精准的对比信息）。

fn format_execution_logs_compact(logs: &[ExecutionLog]) -> String {
    let mut failure_logs = Vec::new();
    let mut success_logs = Vec::new();

    for log in logs {
        // 提取 tool_output 中的错误信息（截断至 150 字符）
        let output_summary = if log.success {
            String::new()
        } else {
            let raw = serde_json::to_string(&log.tool_output).unwrap_or_default();
            let short: String = raw.chars().take(150).collect();
            if raw.chars().count() > 150 {
                format!("{}…", short)
            } else {
                short
            }
        };
        
        let entry = if log.success {
            format!("- ✓ {} ({}ms)", log.tool_name, log.duration_ms)
        } else {
            format!("- ✗ {} ({}ms): {}", log.tool_name, log.duration_ms, output_summary)
        };
        
        if log.success {
            success_logs.push(entry);
        } else {
            failure_logs.push(entry);
        }
    }
    // ... 拼接
}

3. 技能摘要缓存机制
问题：每次 skill_extraction 触发时都需要查询 DB + 反序列化 metadata JSON + 截断 description。如果 2 分钟内触发 3 次（高频失败场景），这些计算是重复的。优化：引入带 TTL 的内存缓存：
use std::collections::HashMap;
use tokio::sync::RwLock;

struct SkillFingerprintCache {
    entries: RwLock<HashMap<String, (Vec<SkillFingerprint>, Instant)>>,
    ttl: Duration,
}

impl SkillFingerprintCache {
    async fn get_or_compute(
        &self,
        space_id: &str,
        store: &MemoryGraphStore,
    ) -> Vec<SkillFingerprint> {
        // 1. 检查缓存
        if let Some((cached, ts)) = self.entries.read().await.get(space_id) {
            if ts.elapsed() < self.ttl {
                return cached.clone();
            }
        }
        // 2. 查询 DB 并计算
        let fingerprints = compute_fingerprints(store, space_id, 50);
        // 3. 写入缓存
        self.entries.write().await.insert(
            space_id.to_string(),
            (fingerprints.clone(), Instant::now()),
        );
        fingerprints
    }
}
TTL 建议 120 秒（与 min_interval_ms 一致），确保同一冷却周期内不会重复查询。
