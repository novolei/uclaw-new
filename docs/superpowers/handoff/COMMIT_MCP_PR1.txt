feat(mcp): wire MCP into agent loop + auto-connect + auto-approve (PR-1)

The 2026-05-18 MCP audit identified that three load-bearing call sites
were unwired: tools never reached the agent, servers never auto-
connected, and the auto_approve flag was inert. This commit closes all
three.

## Backend (src-tauri/src/mcp.rs)

- New `pub const MCP_TOOL_PREFIX = "mcp__"` and two helpers
  `prefixed_tool_name(server, tool)` / `parse_mcp_tool_name(name)`.
  Matches the Cline / Roo / Claude Desktop convention so users see a
  recognizable `mcp__github__create_issue` in tool manifests.
- `McpToolProxy` grows three fields (in addition to existing
  server_id/tool_name/description/schema/manager):
  - `prefixed_name: String` — what `Tool::name()` returns. Keeps the
    raw `tool_name` available for `tools/call` JSON-RPC (must stay
    unprefixed or the server can't route it).
  - `auto_approve: bool` — snapshotted from the source server's
    config at proxy construction time.
- `Tool::requires_approval` override: `Never` when auto_approve=true,
  `UnlessAutoApproved` otherwise. SafetyManager already short-
  circuits Never (safety/mod.rs:221-224), so the flag finally has an
  effect.
- `create_tool_proxies` rewritten to take `&McpManager` (already-
  locked guard) plus the shared handle. Reads connected servers'
  `auto_approve` flags in one pass and emits prefixed-name proxies.
- `McpToolProxy` derives `Clone` so the agent_teams orchestrator's
  sync factory closure can register one proxy per delegate (each
  proxy holds the shared manager Arc — Clone is the natural fit).

## Agent dispatcher wiring (src-tauri/src/tauri_commands.rs)

Three call sites previously sealed `Arc::new(tools)` without ever
asking the MCP manager. Each now calls `McpManager::create_tool_proxies`
(method on the manager, not on the proxy) right before sealing:

1. `send_message` (line ~430): chat IPC. Snapshot via
   `state.mcp_manager.read().await`.
2. `send_agent_message` (line ~8233): agent IPC. Same shape inside
   the spawn-prep block.
3. `start_agent_teams` (line ~11053): teams orchestrator factory
   closure. Snapshot once before `tokio::spawn`, clone per-delegate
   inside the sync factory closure.

## Auto-connect on startup (src-tauri/src/main.rs)

Stage 3 spawn block now fires `mcp_manager.write().await.connect_all_enabled()`
in a fire-and-forget task, right after ImChannelManager startup.
Without this, every app launch required manually clicking "重启" on
each server. Failures are logged + flipped to `Error` status so the
Integrations UI surfaces them.

## Tests (src-tauri/src/mcp.rs#tests)

Five new cases:
- `prefixed_tool_name_format_matches_convention`
- `parse_mcp_tool_name_round_trips`
- `parse_mcp_tool_name_handles_underscore_in_server_id` (the gotcha
  that motivated splitting on `__` not `_`)
- `parse_mcp_tool_name_rejects_non_mcp_names` (builtin names fast-
  path to `None`)
- `create_tool_proxies_emits_prefixed_names_and_honors_auto_approve`
  (end-to-end: build manager, two servers with different auto_approve
  flags, verify each proxy reports the right ApprovalRequirement)

## Verification

Sandbox can't run cargo. Mac side:

```bash
cd src-tauri
cargo build 2>&1 | grep -E "^error" | head
cargo test --lib mcp:: 2>&1 | tail -20
```

Behavioural verification: open Integrations, mark a connected
server's `auto_approve=true`, restart app, observe tools appearing in
the LLM manifest as `mcp__{server}__{tool}` AND the SafetyManager
log line `decision = AutoApprove` rather than `RequireApproval` when
the agent calls one.
