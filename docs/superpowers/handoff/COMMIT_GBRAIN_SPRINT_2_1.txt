feat(gbrain): seed bundled gbrain MCP entry + PGlite data dir (Sprint 2.1)

Sprint 2.0 produced the bundle pipeline (Bun + gbrain source ship as
Tauri resources). This commit closes the runtime gap — at first
launch we seed a default `gbrain` entry into `mcp_servers.json` that
the PR-1 auto-connect hook picks up immediately. After this, the
bundled gbrain spawns as a stdio MCP subprocess on every boot
without any user intervention.

## Backend (src-tauri/src/mcp.rs)

New `McpManager::seed_bundled_gbrain(bun_path, entry_path, pgdata_dir)`.
Idempotent + non-destructive:

- If an entry with id="gbrain" already exists (regardless of
  `enabled`), return Ok(false). Respects user state: removing or
  disabling gbrain in the UI stays sticky across restarts.
- Otherwise build an `McpServerConfig` with:
  - `transport_type = Stdio`
  - `command = <bun>`, `args = [<entry>, "serve"]`
    (matches gbrain's CLI per Sprint 2.0 Mac-side verification)
  - `env = { PGLITE_DATA_DIR: <pgdata_dir> }` so PGlite writes to
    a user-writable path (macOS .app resource dir is read-only)
  - `enabled = true`, `auto_approve = true` (bundled service =
    trusted, same tier as builtin tools)
- Returns Ok(true) on insert, propagates `add_server` errors.

## Path resolution (src-tauri/src/app.rs)

Two new public helpers mirror the existing `find_python` /
`find_bridge_script` pattern:

- `AppState::find_bun_path(resource_dir)` — Tauri resource
  `<res>/bun` first (release), `<CARGO_MANIFEST_DIR>/bunembed/bun`
  fallback (dev). Returns `None` if neither exists.
- `AppState::find_gbrain_entry(resource_dir)` — resource
  `<res>/gbrain/src/cli.ts` first, dev fallback
  `<CARGO_MANIFEST_DIR>/gbrain-source/src/cli.ts`.

Both warn-log if not found. Caller treats `None` as "skip gbrain
seed, app boots normally".

## Boot wiring (src-tauri/src/main.rs)

Stage 3 MCP block now (in order):

1. Resolve `resource_dir_for_mcp` via `app_handle.path().resource_dir().ok()`.
2. Compute `bun_path`, `gbrain_entry` via the new helpers.
3. Compute `pgdata_dir = state_ref.data_dir / "gbrain" / "pgdata"`.
4. `std::fs::create_dir_all(&pgdata_dir)` if bun + entry are both
   present (silent if not — gbrain won't be seeded either).
5. Inside the existing `tokio::spawn` for MCP boot:
   - `set_notification_tx` (PR-4)
   - `set_db_handle` (PR-5)
   - **NEW** `seed_bundled_gbrain` (this commit)
   - `connect_all_enabled` (PR-1)
   - `start_health_loop` for each connected (PR-3)

The seed happens BEFORE `connect_all_enabled` so the new gbrain
entry is part of the same auto-connect pass — first launch goes
straight from "fresh install" to "gbrain running" with no manual
step.

## What the user sees

**First launch after a fresh build:**

- `mcp_servers.json` had no gbrain entry → log line
  "[Stage 3] gbrain MCP entry seeded (first launch)"
- Auto-connect kicks in → log line "Connecting to MCP server
  'gbrain (bundled)'"
- Within seconds: gbrain's tools appear in the agent's manifest
  prefixed `mcp__gbrain__*` (per PR-1's naming)
- `~/.uclaw/gbrain/pgdata/` is populated by PGlite

**Subsequent launches:**

- gbrain entry already in `mcp_servers.json` → seed step is a no-op
- Auto-connect re-establishes the subprocess
- Tools appear in manifest
- PGlite data persists across runs

**User disables gbrain via UI:**

- Entry's `enabled` flag flips to false in `mcp_servers.json`
- Seed step still detects the entry exists, leaves it alone
- Auto-connect skips it (only iterates `enabled=true`)
- User can re-enable via the same toggle later

**User removes gbrain via UI:**

- Entry deleted from `mcp_servers.json`
- Next boot: seed step DOES re-add it (no entry = "fresh install"
  semantics). If the user wants gbrain permanently gone, they
  can also delete the bundle files OR disable instead of remove.
  This is the intentional UX tradeoff — bundled services treat
  "missing" as "we just installed".

## What's NOT done

- **Verify PGLITE_DATA_DIR is gbrain's actual env var name.** Sprint
  2.0 verification didn't read gbrain source directly. If gbrain
  reads a different name (e.g. `GBRAIN_DB_PATH`, `DATABASE_URL`),
  this commit's PGlite path is silently ignored and gbrain falls
  back to its default (probably the cwd, which is the resource
  dir → READ-ONLY → spawn failure or silent data loss). Mac-side
  smoke test post-merge needs to confirm via `ls ~/.uclaw/gbrain/pgdata/`
  after first launch — if empty, the env var name is wrong.
- **Sprint 2.2** — data migration from existing SQLite EntityPages
  into gbrain.
- **Sprint 2.3** — Foundation deprecation roadmap.
- **Sprint 2.4** — Engines spec (Phase 15-21) revision.

## Verification (Mac side)

```bash
cd src-tauri && cargo build 2>&1 | grep -E "^error" | head
cargo tauri dev 2>&1 | tee /tmp/uclaw-boot.log

# Look for the seed + connect lines:
grep -E "gbrain|\[Stage 3\]" /tmp/uclaw-boot.log | head -20

# Confirm PGlite wrote to the right place:
ls ~/.uclaw/gbrain/pgdata/
# Expected: pglite files (postgres data dirs, PG_VERSION, etc).
# If empty → PGLITE_DATA_DIR is the wrong env var name; check gbrain
# source for the correct one and update mcp.rs:seed_bundled_gbrain.

# Confirm gbrain tools reach the agent:
# Open chat / agent → ask "list available tools" or trigger any tool call
# Tools whose name starts with `mcp__gbrain__` are from the bundled gbrain.
```

Cargo build sandbox-clean. Mac side has the final say on the
PGlite env var caveat above.
