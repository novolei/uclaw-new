feat(mcp): auto-reconnect + health loop (PR-3)

Per-server background task that pings every 30s and triggers a
reconnect with exponential backoff on failure. Closes substantive
gap A from the 2026-05-18 audit.

## Loop design (src-tauri/src/mcp.rs)

Three new tunables at module top:
- `HEALTH_PING_INTERVAL_SECS = 30`
- `RECONNECT_INITIAL_DELAY_SECS = 10`
- `RECONNECT_MAX_DELAY_SECS = 300`

`McpManager` grows a `health_tasks: HashMap<String, JoinHandle<()>>`
field for per-server task handles.

Loop body (`run_health_loop`):
1. Sleep `HEALTH_PING_INTERVAL_SECS`, then ping the server.
2. On success: reset the local `attempt` counter to 0, continue.
3. On failure: flip status to Error with a descriptive message
   ("reconnecting in 40s (attempt 3)") so the detail drawer shows
   what's happening, wait `min(INITIAL * 2^attempt, MAX)` seconds,
   call `reconnect_server`. Success resets attempt; failure
   increments and falls through to phase 3 again next iteration.

The reconnect goes through a new internal `reconnect_server` helper
(NOT the public `restart_server`) because the latter aborts the
health loop — calling it from inside the loop would commit suicide.
The internal version inlines disconnect-without-abort + reconnect.

## Lifecycle integration

- `start_health_loop(mgr, id)` (new, public) — idempotent: aborts any
  existing loop for the id, spawns a fresh one. Called from
  `connect_mcp_server` and `restart_mcp_server` Tauri commands plus
  the Stage 3 auto-connect boot pass (after PR-1 added it).
- `stop_health_loop(id)` (new, public) — abort + drop. Called from
  `disconnect_server` and `remove_server`.
- `disconnect_all` now drains the entire `health_tasks` map up
  front, then closes connections; one fewer map op per id during
  app shutdown.

## Boot path (src-tauri/src/main.rs)

The Stage 3 `connect_all_enabled` spawn (added in PR-1) is extended:
after the connect pass, walk `all_server_statuses()` and call
`start_health_loop` for every server now in `Connected` state.
Failed connects are deliberately left without a loop — manual
restart via the UI will spawn one (`restart_mcp_server` IPC
handler).

## Cancellation safety

JoinHandles don't auto-abort when dropped — they detach. Every
remove path (`stop_health_loop`, `disconnect_all`, `remove_server`)
explicitly calls `.abort()`. The boot path doesn't currently abort
loops at app shutdown (uClaw's existing shutdown flow already calls
`disconnect_all`), which handles it transitively.

Race windows considered:
- Loop is mid-reconnect (holding the write lock) when user clicks
  "重启": user's restart waits on the lock, then proceeds normally.
  No correctness issue, mild UX delay.
- User disconnects while loop is sleeping during backoff: stop call
  aborts the JoinHandle, the sleep is canceled cleanly.
- Health loop reconnects while user is reading old status from UI:
  next polled refetch in `IntegrationsModule` picks up the new
  status. Future PR (notifications) will make this push-based.

## What the user sees

- Server connection drops mid-session → 30s later the status badge
  flips to Error with "Health check failed: ... reconnecting in 10s
  (attempt 1)".
- Server comes back → next reconnect attempt succeeds, status flips
  back to Connected, error clears. No manual click required.
- Server stays dead → backoff grows 10s → 20s → 40s → 80s → 160s →
  300s (capped). Each attempt updates the error message so the user
  can see the system is still trying.

## Verification

Sandbox can't run cargo. Mac side:

```bash
cd src-tauri
cargo build 2>&1 | grep -E "^error" | head
cargo test --lib mcp:: 2>&1 | tail -20
```

Manual smoke test:
1. Start an MCP server (any stdio-based example), connect it via UI.
2. Wait 30s — log line `tracing::debug!` should show health ping OK.
3. Kill the server process (`pkill -f mcp-server-...`).
4. Within 30s: status flips to Error, "reconnecting in 10s" message.
5. Restart the server. Next reconnect attempt should succeed.
