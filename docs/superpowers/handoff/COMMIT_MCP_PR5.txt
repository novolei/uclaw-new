feat(mcp): mcp_audit table + env-value redaction + list IPC (PR-5)

Substantive gap C from the 2026-05-18 audit. Three things landing
together because they share the same security envelope:

1. **`mcp_audit` table** (V40 migration) for connection-attempt +
   lifecycle history. Persists across app restarts; the in-memory
   `McpServerState.error` field is still the live source for the
   detail drawer, but the audit log gives users + support a
   reviewable history.
2. **Env-value redaction** in every error string surfaced to UI or
   audit log. A failed subprocess spawn can no longer leak
   `GITHUB_TOKEN=ghp_xxx` in a screenshot.
3. **`list_mcp_audit` IPC** + frontend wrapper so a future "View
   logs" affordance can read the history. UI viewer itself is a
   follow-up.

## Migration (src-tauri/src/db/migrations.rs)

V40_MCP_AUDIT — one table, two indexes:

```
CREATE TABLE mcp_audit (
    id                 TEXT PRIMARY KEY,
    server_id          TEXT NOT NULL,
    event_kind         TEXT NOT NULL,
    message_redacted   TEXT NOT NULL,
    created_at         INTEGER NOT NULL
);
CREATE INDEX idx_mcp_audit_server_time ON mcp_audit(server_id, created_at DESC);
CREATE INDEX idx_mcp_audit_time        ON mcp_audit(created_at DESC);
```

Two indexes for the two read patterns (per-server timeline + global
recent). `message_redacted` is always env-stripped before insert.

## Redaction (src-tauri/src/mcp.rs)

- `pub fn redact_env_values(s: &str, env: &HashMap<String, String>) -> String`
  — replaces any substring matching one of `env`'s values (≥5 chars
  to skip false-positives like `1`/`true`) with `[REDACTED]`. Pure
  function, tested implicitly by the audit-write paths.
- `set_error` now redacts the incoming message using the server's
  own env before storing. So any reads of `McpServerState.error`
  (which feeds the detail drawer's `<pre>` block) get the redacted
  version.
- `McpManager::record_audit` is the wrapper that looks up the
  server's env, redacts, then calls `append_audit_row`.

## Audit hooks

Wired at every significant lifecycle point inside `McpManager`:

- `ConnectAttempt` — before transport spawn
- `ConnectSucceeded` — with tool count
- `HealthFailed` — inside the health loop (PR-3) on each failure
- `Disconnect` — after `disconnect_server`
- `Removed` — before `remove_server` drops state

Additional kinds defined for future use: `ConnectFailed`,
`Reconnected`, `ToolsChanged`. Wiring them is one-line per call
site.

## DB plumbing

`McpManager` grows an `Option<Arc<Mutex<Connection>>>` field and a
`set_db_handle` method. Wired at Stage 3 boot in `main.rs` —
alongside the PR-4 notification channel setup, the same spawn pulls
`state_ref.db` into `db_for_mcp` and hands it to the manager.

When `db` is `None` (unit tests don't install one), every audit
write is a silent no-op. This keeps the existing mcp tests passing
without DB scaffolding.

## IPC (src-tauri/src/tauri_commands.rs)

`list_mcp_audit(server_id: Option<String>, limit: Option<u32>) -> Vec<McpAuditEntry>`

- `server_id = None` → all rows, most recent first.
- `limit` clamped to [1, 1000], default 100.
- Returns `McpAuditEntry` (mirror of the DB row, with camelCase via
  serde rename_all on Rust side; the type is exported from
  `mcp.rs` so both `tauri_commands` and `ipc` modules can use it).

Frontend wrapper + TypeScript type added to `ui/src/lib/tauri-bridge.ts`.

## Limitations / non-goals

- **No UI viewer yet.** The audit table is populated and queryable;
  a future small PR can add a "View logs" tab to the detail drawer
  + an Integrations-module overview.
- **No raw-logs opt-in.** The audit always redacts. A
  `memubot_config.mcp.raw_logs_enabled` flag could add a path that
  inserts the unredacted message; the spec'd structure is there
  but the flag isn't (would need a clear "this can leak secrets"
  consent step).
- **Pruning.** No TTL on rows yet. A typical user might accumulate
  a few hundred per day at most (one row per connect/disconnect
  cycle); the indexes keep reads fast. A future TTL job can prune
  rows older than N days.
