feat(mcp): server notification routing — tools/list_changed (PR-4)

The stdout reader task previously logged-and-discarded server-pushed
JSON-RPC notifications (mcp.rs:362-369). Substantive gap B from the
audit. This commit routes them through an mpsc channel to a manager-
side consumer that dispatches by method; the canonical
`notifications/tools/list_changed` triggers an auto-refresh + Tauri
event that the Integrations module listens for.

## Plumbing

Stdio reader → mpsc::UnboundedSender → consumer task → manager.refresh_tools
                                                       → `Emitter::emit("mcp:tools-changed", ...)`

### Types (src-tauri/src/mcp.rs)

- `pub struct McpNotificationEvent { server_id, method, params }` —
  the wire shape for forwarded notifications. `params` is `serde_json::Value`
  so future spec additions (logging, sampling, progress) can be added
  without re-touching this struct.
- `pub const NOTIFY_TOOLS_LIST_CHANGED = "notifications/tools/list_changed"`.

### McpManager

- New `notification_tx: Option<UnboundedSender<McpNotificationEvent>>`
  field, installed via `set_notification_tx()` at boot.
- `connect_server` now passes the sender into `StdioTransport::spawn`.
  HTTP transport ignores it (request-response only; no server push
  channel until SSE support lands).

### StdioTransport::spawn

Signature gains a `server_id` and an optional sender. The reader task
clones both; when it sees a JSON-RPC frame with `method` it builds an
`McpNotificationEvent` and `tx.send(event)`. Send failure (receiver
dropped) is logged but doesn't kill the reader — the read side of the
protocol stays correct even if the consumer task died.

### Consumer (src-tauri/src/main.rs)

Stage 3 spawn block grows a consumer task that drains the receiver
for the lifetime of the app:

- `event.method == NOTIFY_TOOLS_LIST_CHANGED` → call
  `mgr.refresh_tools(server_id)`, emit Tauri event
  `"mcp:tools-changed"` with `{ serverId, toolCount }`.
- Anything else → `tracing::debug!` (keeps observability for the
  spec's other notification methods without breaking forward-compat).

The channel sender is installed BEFORE the `connect_all_enabled` pass
so any list_changed fired during the initial handshake is captured.

## Frontend (ui/.../IntegrationsModule.tsx)

`useEffect` subscribes to `mcp:tools-changed` via `@tauri-apps/api/event`.
On event: call existing `refetch()`. Idempotent + cheap (it's just
two parallel `list_*` IPC calls). The listen promise resolves to an
unsubscribe handle that the cleanup arm calls. Subscribe failure
falls back to the existing polling-on-action UX — not a hard error.

## What the user sees

- Connect an MCP server that adds tools after `initialize` (some
  github-style servers register tools lazily after auth).
- Wait — no manual refresh needed. New tools appear in the card +
  detail drawer within milliseconds of the server pushing
  `notifications/tools/list_changed`.

## Limitations / non-goals

- HTTP transport: still no notification path. The MCP spec covers SSE
  for HTTP push but uClaw doesn't wire that yet. Future PR can layer
  it onto this same channel/consumer.
- Other notification methods (`notifications/resources/list_changed`,
  `notifications/prompts/list_changed`, sampling/logging requests):
  recognized in the consumer's fallback `else` branch, logged at
  debug. Each becomes a one-liner pattern arm when its consumer is
  written.
- The consumer task never shuts down explicitly — it ends when the
  channel is dropped, which happens at app exit. Sufficient for
  current shutdown flow; a future shutdown signal could break the
  recv loop earlier if needed.

## Verification

Sandbox: `npx tsc --noEmit` clean. Mac side cargo + manual test
needed.

Smoke test:
1. Start a stdio MCP server that supports tools/list_changed
   (e.g. the official @modelcontextprotocol/server-everything has a
   notification trigger via its CLI).
2. Connect via UI, observe tool list.
3. Trigger the server to add a tool. Within ~1s the card's tool count
   should update without any user action.
