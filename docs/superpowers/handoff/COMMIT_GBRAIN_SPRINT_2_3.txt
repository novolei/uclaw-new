feat(gbrain): agent system prompt instruction block (Sprint 2.3)

gbrain MCP is alive (Sprint 2.0-2.2.5b) and the agent has
`mcp__gbrain__*` tools in its manifest. But the system prompt has
zero instructions about when to call them — the LLM has no
incentive structure to put_page or recall. End result: gbrain stays
"alive but idle", same dead-code pattern Foundation Phase 1-7's
EntityPage UI fell into.

This commit gives the LLM explicit guidance. Mirrors Sprint 1.6's
`UserProfileSection` pattern: a `render` function returns
`Option<String>` based on whether the dependent tools are
currently visible.

## Module (src-tauri/src/agent/gbrain_prompt.rs)

New `GbrainKnowledgeSection::render(mcp_mgr: &McpManager)`:
- Returns `None` when no `mcp__gbrain__*` tools registered (server
  disconnected, bundle missing, init failed) — caller skips append
  so LLM never sees instructions for tools that don't exist
- Returns `Some` with a ~50-line markdown block explaining when to
  call `put_page` / `recall`, slug format, content format, what to
  do after the call, and what NOT to do
- Block starts with `## Long-term Knowledge (gbrain)` marker so
  tests + logs can grep without depending on prose

3 unit tests cover:
- `render_returns_none_when_no_gbrain_tools`
- `render_returns_some_with_marker_when_gbrain_connected`
- `render_returns_none_when_other_mcp_servers_but_no_gbrain` (a
  github server doesn't trigger gbrain section — forward-compat)

## Dispatcher wiring (src-tauri/src/agent/dispatcher.rs)

`ChatDelegate` gains a `gbrain_knowledge_block: String` field
(mirrors `learned_profile_block` shape). `set_gbrain_knowledge_block`
setter. `effective_system_prompt` appends the block right after
the learned-profile block — both live in the dynamic-context
section that's regenerated per turn, NOT in the cached system
prompt, so tool-presence changes mid-session take effect on the
next prompt without restarting the agent.

## Call sites (src-tauri/src/tauri_commands.rs)

Three `ChatDelegate::new` paths now build + inject the block:

1. `send_message` (chat IPC): direct `state.mcp_manager.read().await`
   → render → setter. Sync call, no Arc juggling.
2. `send_agent_message` (agent IPC): pre-renders before
   `tokio::spawn` (where `state.mcp_manager` is still accessible),
   passes the String through. Same shape as Sprint 2.0's
   `learning_buffer_for_spawn`.
3. `start_agent_teams` (factory closure): pre-renders next to the
   existing `mcp_proxies_for_factory` snapshot — both consume the
   same MCP manager read guard. Cloned per delegate inside the
   sync `Fn` closure.

For paths 2 and 3, the snapshot semantics match Sprint 2.0/PR-1:
a gbrain reconnect mid-session won't update the in-flight delegate;
the next message creates a new delegate with the new state.

## Verification

```bash
cd src-tauri
cargo build 2>&1 | grep -E "^error" | head
cargo test --lib agent::gbrain_prompt 2>&1 | tail
```

Behavioural test post-merge:

```
1. Restart uClaw. Confirm Settings → 系统 → gbrain init = Succeeded.
2. Start new chat: "OpenAI released GPT-5 in May 2026."
3. Look at agent_messages — agent should call mcp__gbrain__put_page
   with slug like "openai-gpt-5-release" and YAML-frontmatter content.
4. Restart uClaw.
5. New chat: "Do you remember GPT-5?"
6. Agent should call mcp__gbrain__recall and cite the slug.
```

If the agent doesn't call gbrain tools even after this lands, the
issue is one layer up — either gbrain isn't actually in the tool
table (check Integrations UI / list_mcp_tools IPC), or the LLM's
system prompt cache is masking the new section (try a "Clear chat
context" reset). Both are diagnosable from the Settings → 系统 tab.
