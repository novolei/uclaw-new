## 0. Why these three together

Sprint 2.2.x landed the gbrain infrastructure: bundle pipeline (2.0),
seed + init (2.1 + init-fix), launcher + recovery script (2.2). PR
#208 fixed the write-lock-during-IO deadlock. PR #206 added a system
diagnostics tab. PR #214/#216 added `/v1/embeddings` so gbrain stays
fully local.

But agent doesn't USE gbrain — the tools `mcp__gbrain__put_page`,
`mcp__gbrain__recall`, `mcp__gbrain__extract_facts` are registered
in the agent's tool table but the system prompt has zero instructions
telling the LLM **when** to call them. End result: gbrain is "alive
but idle" — the Foundation Phase 1-7 wiki dead-code pattern repeating.

These three sprints together close the loop:

| Sprint | Closes | Risk |
|---|---|---|
| **2.2.5** — 3 reliability bugs | gbrain init can hang boot, error messages aren't actionable, embedding URL silent failure | Low |
| **2.3** — Agent prompt section | Agent never calls gbrain because nothing instructs it | Low |
| **2.4** — Chat → gbrain auto-extractor | Even with instructions, agent only learns reactively; need proactive ingestion | Medium |

Single branch, 6 bisectable commits, single PR.

### Commit 1 — Sprint 2.2.5a: timeout on gbrain init

**Problem:** `ensure_bundled_gbrain_initialized` (mcp.rs:1957) spawns
`bun gbrain/cli.ts init --pglite --yes` via `std::process::Command`
synchronously. If bun corrupts / PGLite deadlocks / I/O hangs, the
whole uClaw boot hangs with no recourse except force-kill.

**Fix:** Wrap the spawn in a tokio timeout. Since the existing call
site is `&mut self` (sync), we need to either:

- Option A: Convert `ensure_bundled_gbrain_initialized` to async and
  use `tokio::time::timeout(Duration::from_secs(120), ...)`. Caller
  in main.rs Stage 3 is already async-spawn context.
- Option B: Stay sync but use `wait_timeout` crate.

**Decision:** **Option A** (async). Cleaner, no new dep. The sync
signature was an accident of how seed_bundled_gbrain was originally
sync — but init is naturally an async operation since it spawns a
subprocess we want to bound on time.

**Files touched:**
- `src-tauri/src/mcp.rs` — convert `ensure_bundled_gbrain_initialized`
  to async fn, use `tokio::process::Command` + `tokio::time::timeout`
- `src-tauri/src/main.rs` Stage 3 — call site already in async
  context, just await

**Constants:**

```rust
/// Sprint 2.2.5 — Max wall-clock for first-launch gbrain init.
/// PGLite migrations should fit in ~30s on cold disk; 120s is the
/// "something went very wrong" ceiling. After timeout, init is
/// considered failed (caller treats as graceful degradation: app
/// boots without gbrain).
pub const GBRAIN_INIT_TIMEOUT_SECS: u64 = 120;
```

**Tests:** Add `#[tokio::test]` that spawns a `sleep 300` instead of
real bun + asserts that the timeout fires within 130s. Use a fake
binary path to keep the test fast and deterministic.

**Commit message template:** `COMMIT_GBRAIN_2_2_5_A.txt` (see §3).

### Commit 2 — Sprint 2.2.5b: store init error in AppState + surface via diagnostics

**Problem:** When init fails, the error is logged to tracing but
never surfaced to the user. `restart_gbrain_mcp` returns "entry not
seeded" with no hint to run `scripts/init-gbrain.sh` or check logs.

**Fix:** Store the last init outcome in `AppState`. Existing
`gbrain_mcp_id: Arc<Mutex<Option<String>>>` field (added in PR #206)
gets a sibling:

```rust
pub gbrain_init_status: Arc<Mutex<GbrainInitStatus>>,
```

where:

```rust
#[derive(Debug, Clone, Serialize)]
#[serde(tag = "status")]
pub enum GbrainInitStatus {
    NotAttempted,                                  // app booted but never tried (e.g. bundle missing)
    InProgress,                                    // init currently running
    Succeeded { duration_ms: u64, at_ms: i64 },
    Failed { error: String, stderr_tail: Option<String>, at_ms: i64 },
    SkippedAlreadyInitialized { at_ms: i64 },      // PG_VERSION already existed
    BundleMissing,                                 // find_bun_path / find_gbrain_entry returned None
}
```

`ensure_bundled_gbrain_initialized` updates this slot before returning.
`get_system_diagnostics` includes it in the response.

The frontend SystemTab (PR #206) gets a small new section showing
gbrain init status with friendly text + an actionable "Run init
manually" hint when `Failed`.

**Files touched:**
- `src-tauri/src/app.rs` — new field on AppState, new enum type
- `src-tauri/src/mcp.rs` — `ensure_bundled_gbrain_initialized` writes
  the status slot (caller passes a handle)
- `src-tauri/src/main.rs` — pass the Arc into the init call
- `src-tauri/src/tauri_commands.rs` — `SystemDiagnosticsReport`
  struct gains `gbrain_init: GbrainInitStatus`, populated in
  `get_system_diagnostics`
- `ui/src/lib/types.ts` — TS mirror of `GbrainInitStatus`
- `ui/src/components/settings/SystemTab.tsx` — new status row

**Tests:** Backend unit test that runs `ensure_bundled_gbrain_initialized`
with a non-existent bun path → expects `Failed` in the slot.

### Commit 4 — Sprint 2.3: agent system prompt gbrain section

**Problem:** Agent has `mcp__gbrain__*` tools but no instructions on
when to use them. They're invisible to the LLM's reasoning.

**Fix:** New module `src-tauri/src/agent/gbrain_prompt.rs` that
renders a system-prompt block when gbrain MCP is connected:

```rust
pub struct GbrainKnowledgeSection;

impl GbrainKnowledgeSection {
    /// Render the gbrain instruction block. Returns `None` when
    /// gbrain isn't currently connected (no tools registered yet
    /// or server in Error state). Caller skips injection in that
    /// case so the prompt doesn't reference tools that don't exist.
    pub fn render(mcp_mgr: &McpManager) -> Option<String> {
        let has_gbrain = mcp_mgr
            .all_tools()
            .iter()
            .any(|t| t.server_id == "gbrain");
        if !has_gbrain {
            return None;
        }
        Some(GBRAIN_INSTRUCTIONS.to_string())
    }
}

const GBRAIN_INSTRUCTIONS: &str = "## Long-term Knowledge (gbrain)

You have a persistent local knowledge base via `mcp__gbrain__*` tools.
gbrain is a wiki-style entity graph backed by PGLite. It survives
across conversations and uClaw restarts. Use it PROACTIVELY:

When to call `mcp__gbrain__put_page`:
- User introduces a new entity worth long-term retention (a person,
  company, project, concept, decision, claim)
- User explicitly asks you to remember (\"记住\", \"remember this\")
- Conversation surfaces a stable fact (e.g. \"GPT-5 released in 2026\")
- A multi-turn investigation reaches a conclusion worth preserving

When to call `mcp__gbrain__recall`:
- User asks \"do you remember\" / \"what did we say about\"
- A new question echoes a topic from a prior session
- Before answering a factual question that gbrain might know

Slug format: kebab-case English, namespaced when useful. Examples:
- \"openai-gpt-5-release\"
- \"project-uclaw\"
- \"decision-trigram-fts-2026-05\"

Content format: YAML frontmatter (title, type, aliases, tags) +
markdown body. Keep bodies under 500 words; link to sub-pages via
`[[other-slug]]` when content grows.

After calling `recall`, cite the slug(s) you read from in your
response. After calling `put_page`, mention you've recorded it.

DO NOT:
- Put `put_page` calls in your visible response — chain them as
  separate tool calls
- Use `put_page` for ephemeral things (this turn's question, jokes)
- Call `recall` for general-knowledge questions that don't reference
  prior conversation
";
```

Wire into `ChatDelegate::effective_system_prompt` after existing
sections. The dispatcher already has access to `state.mcp_manager`
in the IPC handlers — needs to be passed into the delegate.

**Files touched:**
