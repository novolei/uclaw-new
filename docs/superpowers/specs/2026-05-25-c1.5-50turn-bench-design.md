## 1. Background

### 1.1 Why this exists

The 8-PR Dirac borrow sequence (A1–A4, B1, B2) shipped token-saving
capabilities whose savings were only ever **modeled**, never measured:

- **A2** multi-file `edit{files:[...]}` — collapses N single-file edits into 1.
- **A3** `[File Hash:]` + `assume_hash` — short-circuits repeated reads.
- **B1** word-anchor edits — eliminates `old_text` drift / re-read churn.
- **B2** ContextManager wire-up — context-on-demand (fragment selection).

A live diagnostic (a trivial "在吗" turn) confirmed the gap empirically:
its 16,685 input tokens were ~60% conversation history + ~41% the 32-tool
schema set, with **zero** Dirac savings (no file ops) and a small
unconditional **cost** (B2 added 2 context tools to the schema). The borrows
only pay off on **file-heavy refactor tasks** — exactly what this bench drives.

### 1.2 What telemetry already exists (no new measurement plumbing)

`agent::dispatcher` already logs, per LLM call:

- `Calling LLM ... system_prompt_tokens=N tool_def_tokens=N message_tokens=N
  message_count=N tool_count=N estimated_total=N`
- `on_usage input_tokens=N output_tokens=N cache_read_tokens=N
  cache_creation_tokens=N cost_usd=$X` (the `agent:turn_cost` event)
- `agent:context_stats` (model_context_length, used, free)

The bench **harvests these existing signals**; it does not add a new metering
layer. The same tokenizer the app uses is used for both pre/post builds, so
comparisons are apples-to-apples.

### 1.3 M2 DoD targets being quantified

From `uclaw-upgrade-implementation-plan.md` §4.3 (M2 完成定义):

| DoD gate | Target |
|---|---|
| 50-turn session token reduction | **−60–75%** |
| Cached-token hit rate | **≥50%** |
| Monthly cost reduction | **↓60%** |
| Output format consistency (subjective) | **+1.5/5** |

## 2. Scope

A bench binary + golden fixtures + an orchestration script. No changes to
the agent runtime itself.

### 2.1 In scope

1. **Golden fixture** `src-tauri/tests/fixtures/c1.5-bench/refactor-8-file/`:
   an 8-file workspace with a known symbol (`fn old_name`) to rename to
   `new_name` across all files, plus two pre-recorded **golden tool-call
   sequences** (`pre_dirac.jsonl`, `post_dirac.jsonl`) encoding how an agent
   without vs with the borrows performs the refactor.
2. **`MockLlm`** deterministic replay provider: returns the next tool-call
   from a golden sequence instead of calling a network LLM. The dispatcher
   still builds the **real** request each turn (real system prompt + real
   tool defs + accumulating history); only the LLM *response* is mocked.
3. **Bench binary** `src-tauri/src/bin/bench_50turn.rs`:
   `--fixture <name> --mode replay|live --provider deepseek|anthropic
   --runs K --out <json>`. Replay mode is the gating, deterministic path;
   `--mode live` does real provider calls for adoption + cache confirmation.
4. **Orchestration script** `scripts/bench/c1.5-50turn.sh`: stands up a
   pre-Dirac baseline build via `git worktree`, copies the bench binary +
   fixtures into it, builds both binaries, runs replay on each, diffs, and
   (optionally, when API keys are present) runs live mode, then writes the
   report.
5. **Report** `docs/superpowers/reports/2026-05-25-M2-benchmark.md`: per-metric
   pre/post table + a met/not-met/N-A verdict against each M2 DoD gate, with
   explicit caveats.
6. A small unit test that the golden sequence + `MockLlm` produce stable token
   counts (so the bench itself doesn't silently drift).

### 2.2 Out of scope

- **Automated format-consistency scoring** — subjective; ships as a **manual
  rubric appendix** (rate 5 sample outputs pre/post on a 5-point scale), NOT
  an automated gate.
- **Monthly cost** — a production-usage metric; the bench reports **per-task
  cost delta** as a documented proxy.
- **A1 (pairing repair) on deepseek** — A1 is Anthropic-specific; the bench
  marks it N/A on deepseek and validates it only on Anthropic.
- **Changing the agent runtime / adding feature flags** — the baseline is
  obtained by building the real pre-Dirac commit, not by flagging borrows off.
- **CI integration** — the bench is a manual/on-demand tool for this
  measurement; wiring it as a recurring CI gate is a possible follow-up, not
  part of C1.5.

## 3. Design

### 3.1 Measurement philosophy: hybrid (deterministic replay + live confirmation)

Two complementary measurements:

- **Deterministic replay (gating)**: a fixed golden tool-call sequence is
  replayed against the **pre-Dirac** and **post-Dirac** builds. Because the
  tool-call sequence is fixed, the only variables are (a) tool-schema sizes
  (A2 batch / B1 anchored / A3 read description / B2 context tools), (b)
  baseline-block composition (A4), and (c) **round-trip count** (pre does the
  refactor in ~2N turns; post in ~2–3). Free, reproducible, no network.
- **Live confirmation (non-gating)**: a real refactor *prompt* (not scripted
  tool calls) is sent to deepseek + Anthropic on the **post** build, K=3 runs
  each, to (a) confirm the LLM **actually adopts** the borrows (emits
  `edit{files:[]}` / `assume_hash` / anchored edits), and (b) measure real
  `cache_read_tokens`. This validates the replay's load-bearing assumption
  ("the LLM uses the new capabilities").

The replay produces the headline numbers; the live runs keep them honest.

### 3.2 Golden fixture + sequences

`src-tauri/tests/fixtures/c1.5-bench/refactor-8-file/`:

- `workspace/` — 8 small source files (`mod_a.rs` … `mod_h.rs`), each
  containing a call to / definition of `old_name`. Total ~8×40 lines.
- `pre_dirac.jsonl` — the legacy behavior: for each file, `read_file{path}`
  then `edit{path, edits:[{old_text:"old_name", new_text:"new_name"}]}` →
  **16 tool calls / ~16 turns**.
- `post_dirac.jsonl` — the borrow behavior: one `read_file{path}` per file
  reusing `assume_hash` on the second pass (A3), then ONE
  `edit{files:[{path,edits:[anchored…]}×8]}` (A2 + B1) → **~3 tool calls /
  ~3 turns**.
- `task.json` — the user prompt ("rename `old_name` to `new_name` across the
  workspace") used by live mode, plus the expected final workspace state for
  a correctness assertion.

Each golden record is `{turn, tool_name, tool_args, mock_tool_result}`. The
mock tool result is the deterministic stand-in the dispatcher would have
produced (so history grows realistically without touching the real FS beyond
the fixture workspace).

### 3.3 `MockLlm` provider

A test-only LLM provider implementing the same `LlmProvider`-shaped interface
the dispatcher calls. On each `complete(request)`:

1. Records the **request token breakdown** the dispatcher computed
   (system_prompt / tool_def / message / total) — this is the measurement.
2. Returns the next golden record's tool-call as the assistant message
   (or a terminal text message when the sequence is exhausted).

`MockLlm` never hits the network → deterministic + free. It lives behind a
`#[cfg(...)]`/bench-only module so it is not compiled into the shipping app.

### 3.4 Bench binary `src-tauri/src/bin/bench_50turn.rs`

```
bench_50turn --fixture refactor-8-file --mode replay --out post.json
bench_50turn --fixture refactor-8-file --mode replay --golden pre  --out base.json   # for the baseline build
bench_50turn --fixture refactor-8-file --mode live --provider deepseek --runs 3 --out live-ds.json
bench_50turn --fixture refactor-8-file --mode live --provider anthropic --runs 3 --out live-an.json
```

- **replay**: builds a `ToolRegistry` + composes the system prompt via the
  build's own `compose_system_prompt*` + plays the golden sequence through a
  dispatcher driven by `MockLlm`, summing per-turn token breakdowns →
  `{turns, total_input_tokens, total_round_trips, per_turn:[…]}` JSON.
- **live**: builds the same registry + sends the `task.json` prompt to the
  real provider, K runs; harvests `on_usage` (input/output/cache_read/cost)
  and records adoption flags (did the assistant emit batch / anchored /
  assume_hash). Writes averaged JSON.

The binary depends only on interfaces present in BOTH pre- and post-Dirac
builds (`ToolRegistry::new`, `compose_system_prompt`, the tokenizer, the
dispatcher's request-build path). Where the post build renamed/extended an
interface (e.g. B2's `compose_system_prompt_with_injection`), the bench calls
the lowest-common-denominator entry (`compose_system_prompt`) which exists in
both, guarded by `#[cfg(...)]` only if a signature genuinely diverged.

### 3.5 Orchestration script `scripts/bench/c1.5-50turn.sh`

```
1. BASE_SHA=f6447a71   # last commit before A1 (#494); pre-Dirac
2. git worktree add ../uclaw-bench-baseline $BASE_SHA
3. cp src-tauri/src/bin/bench_50turn.rs            ../uclaw-bench-baseline/src-tauri/src/bin/
   cp -r src-tauri/tests/fixtures/c1.5-bench        ../uclaw-bench-baseline/src-tauri/tests/fixtures/
   # + register the bin in the baseline Cargo.toml if needed
4. (cd ../uclaw-bench-baseline/src-tauri && cargo build --bin bench_50turn)   # baseline
   (cd src-tauri && cargo build --bin bench_50turn)                          # post
5. baseline replay  → base.json ; post replay → post.json
6. diff → deltas (token −%, round-trip −%, cost −%)
7. if ANTHROPIC_API_KEY / DEEPSEEK_API_KEY present: post live runs → live-*.json
8. render docs/superpowers/reports/2026-05-25-M2-benchmark.md
9. git worktree remove ../uclaw-bench-baseline
```

The baseline worktree is created from `f6447a71`. **Risk**: that commit
predates this bench, so step 3 forward-ports the bench artifacts into it. If
the pre-Dirac `compose_system_prompt` / `ToolRegistry` signatures differ from
post in a way the copied bench can't bridge, the script aborts with a clear
message and the bench falls back to documenting the per-build numbers it could
collect (it never silently emits a bogus delta).

### 3.6 Metrics → DoD verdict

| Metric | Source | DoD gate | Verdict rule |
|---|---|---|---|
| Σ input tokens (whole scenario) | replay | −60–75% | met if `(base−post)/base ∈ [0.60, 0.75]`+ |
| round-trip count | replay | (A2/B1 headline) | report ratio (16→~3 expected) |
| cache-hit rate | live | ≥50% | met if `cache_read/input ≥ 0.50` on ≥1 provider |
| per-task cost | live (+ replay est.) | ↓60% (proxy for monthly) | met if `(base−post)/base ≥ 0.60` |
| format consistency | **manual rubric** | +1.5/5 | appendix; not auto-gated |
| borrow adoption | live | — | pass if LLM emits batch+anchored+assume_hash |

## 5. Tests

| # | Test | Scenario |
|---|---|---|
| 1 | `golden_sequence_replays_deterministically` | replay the post golden twice → identical per-turn token breakdown |
| 2 | `pre_golden_has_more_roundtrips_than_post` | pre sequence turn-count > post sequence turn-count (the structural premise) |
| 3 | `mock_llm_returns_golden_calls_in_order` | `MockLlm` yields the golden tool-calls in sequence then terminates |
| 4 | `replay_emits_per_turn_token_breakdown` | replay output JSON has system/tool/message token fields per turn |
| 5 | `fixture_workspace_final_state_matches_expected` | applying the post golden edits yields `task.json`'s expected workspace |

Live mode is exercised manually (needs API keys) — not a unit test.

```bash
cd src-tauri && cargo test --test c1_5_bench 2>&1 | tail -10                    # 5 passing
cd src-tauri && cargo run --bin bench_50turn -- --fixture refactor-8-file --mode replay --out /tmp/post.json
# inspect /tmp/post.json: total_round_trips ~3, per-turn token breakdown present
bash scripts/bench/c1.5-50turn.sh   # full pre/post + report (replay always; live if keys present)
```

Acceptance: the script produces `docs/superpowers/reports/2026-05-25-M2-benchmark.md`
with concrete pre/post numbers and a verdict per DoD gate. If the −60–75%
token target is NOT met, that is a **valid, reportable result** (it informs
whether the M2 DoD is actually achieved), not a bench failure.

## 7. Migration / rollback

- **DB migration**: none.
- **Backward compat**: bench-only; no shipping code touched. `MockLlm` and the
  bench bin are excluded from release builds.
- **Rollback**: delete the bench bin + fixtures + script + report. No runtime
  impact.

## 8. Decisions (locked 2026-05-25)

### 8.1 Hybrid measurement (replay gating + live confirmation)
- **Why**: replay gives free, reproducible, apples-to-apples token/round-trip
  deltas; live confirms the LLM actually adopts the borrows (without which the
  replay's premise is unproven). Pure replay would beg the adoption question;
  pure live is non-deterministic + costs money.

### 8.2 Baseline via real pre-Dirac build (`git worktree` @ f6447a71)
- **Why**: measures the **real** pre vs post code, not a feature-flagged
  approximation (the borrows weren't built with flags; adding them would be
  invasive and itself perturb the numbers). Accepts the cost of building two
  binaries + forward-porting the bench into the baseline worktree.

### 8.3 Provider scope: deepseek + Anthropic
- **Why**: deepseek-v4-pro is the user's real environment (validates A2/A3/B1/B2
  + deepseek prefix cache); Anthropic is the only place A1 (pairing repair) and
  `cache_control: ephemeral` (B2 byte-stability's payoff) can be validated.

### 8.4 Format-consistency is a manual rubric, not an auto-gate
- **Why**: it is subjective (+1.5/5 on output quality); automating it reliably
  is its own research problem. Ships as an appendix the human fills in by
  rating 5 sample outputs pre/post.

### 8.5 8-file rename refactor as the canonical fixture
- **Why**: a rename is the cleanest multi-file refactor — deterministic
  expected output, exercises read (A3), multi-file batch edit (A2), and
  anchored targeting (B1) together, and maps to Dirac's own `refactor_*` evals.

## 12. Autonomous execution mode

When executed via the autonomous orchestrator
([`protocols/autonomous-execution-protocol.md`](../protocols/autonomous-execution-protocol.md)):

- **Reviewer focus**: (a) replay is genuinely deterministic (test #1);
  (b) the baseline build is the real pre-Dirac commit, not a stub; (c) the
  report NEVER emits a delta it couldn't actually measure (no fabricated
  numbers — §3.5 abort path); (d) `MockLlm`/bench bin are excluded from release
  builds.
- **No-fabrication invariant**: if live runs can't execute (no API keys), the
  report says so explicitly and emits only replay numbers — it must not invent
  cache-hit or cost figures.
- **CI**: the deterministic tests (#1–#5) can gate; the live mode + the
  cross-build script are manual (need keys + two builds).
- **Risk class**: LOW (bench-only, no runtime change) — but the *numbers it
  produces* are decision-grade, so the harness's honesty (no fabricated
  deltas) is the property under review.
