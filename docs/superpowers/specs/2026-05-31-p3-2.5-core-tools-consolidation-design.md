# P3-2.5 — Core-Tool Construction Consolidation Design

**Date:** 2026-05-31
**Status:** Design (approved in brainstorming; pending spec review)
**Strategic alignment:** CLAUDE.md open debt "collapse 4 registries → one handle"; Pi-lightweight philosophy ADR (`2026-05-28-uclaw-pi-lightweight-product-philosophy.md`) — one seam, no over-engineering.

## Problem

#610's descriptor migration (P3-2) moved the 9 core workspace tools (ReadFile, WriteFile, GetFileSkeleton, Grep, Glob, WebFetch, HttpRequest, Edit, Bash) to descriptor-based construction for the **main session** (`build_tool_registry` → `AgentApi::build_session_registry`). But the **AgentTeamOrchestrator `delegate_factory`** is a separate **sync** closure that hand-registers a curated subset of the same core tools. `builtin_descriptors.rs` itself notes: "Tools not migrated here are still registered inline by `build_tool_registry()` (P3-2.5 follow-up will complete the migration)."

The concrete cost surfaced while landing item 2/3: per-tool config (`EditTool::with_project_check`, `ReadFileTool::with_max_read_chars`) had to be wired in **two places** — the descriptor builders AND the team closure — because the core-tool construction logic is duplicated.

### Why the duplication is real (not laziness)

The `delegate_factory` is `move |system_prompt: String| -> Box<dyn LoopDelegate + Send>` — a **sync** closure that deliberately does **not** hold `&AppState`. Every dependency is pre-extracted into `*_for_factory` locals before the surrounding `tokio::spawn(async move)` because `&AppState` is neither `'static` nor freely movable into the closure. `AgentApi::build_session_registry(&ctx)` is sync, but its `SessionContext` requires `&AppState` — which the closure cannot supply. So the closure cannot call the descriptor path. This `&AppState` lifetime/Send boundary is the structural reason the two paths diverged.

## Goal

Make the 9 core tools have a **single sync construction source** that both paths call, so per-tool config flows through **one seam**. Do NOT migrate the legitimately-async tools (browser/skill/memu/plugins) to descriptors (they need `state.settings.read().await`, `ctx_mgr...await`, etc.) — that is out of scope (the rejected "full P3-2.5 async-descriptor" option). Do NOT decouple `SessionContext` from `&AppState` (the rejected "true one-handle" option, higher risk).

## Design

### New module: `src-tauri/src/agent/tools/core_tools.rs`

The single, sync, `&AppState`-free construction point for the 9 core workspace tools.

```rust
/// The one per-tool config seam. Adding a new per-tool knob means adding a
/// field here + applying it in the matching constructor below — nowhere else.
#[derive(Debug, Clone)]
pub struct ToolConfig {
    /// Some → enable best-effort post-edit project check on EditTool.
    pub edit_project_check: Option<crate::agent::tools::builtin::edit_verify::ProjectCheckCfg>,
    /// read_file truncation cap (floor-clamped in the tool). Default MAX_READ_CHARS.
    pub read_file_max_chars: usize,
}

impl Default for ToolConfig {
    fn default() -> Self {
        Self {
            edit_project_check: None,
            read_file_max_chars: crate::agent::tools::builtin::file::MAX_READ_CHARS,
        }
    }
}

// Per-tool constructors — config wiring lives ONLY here.
pub fn edit_tool(ws: &Path, cfg: &ToolConfig) -> builtin::edit::EditTool;        // applies with_project_check
pub fn read_file_tool(ws: &Path, cfg: &ToolConfig) -> builtin::file::ReadFileTool; // applies with_max_read_chars
pub fn write_file_tool(ws: &Path) -> builtin::file::WriteFileTool;                // plain
pub fn get_file_skeleton_tool(ws: &Path) -> ...;                                  // plain
pub fn grep_tool(ws: &Path) -> ...;                                               // plain
pub fn glob_tool(ws: &Path) -> ...;                                              // plain
pub fn web_fetch_tool() -> ...;                                                  // plain
pub fn http_request_tool() -> ...;                                               // plain
pub fn bash_tool(ws: &Path) -> ...;                                              // plain

/// Batch helper — registers all 9 core tools with config applied. One-line for callers.
pub fn register_core_tools(reg: &mut ToolRegistry, ws: &Path, cfg: &ToolConfig);
```

`register_core_tools` internally calls the per-tool constructors. Both granularities exist: per-tool fns for the descriptor builders (which register one tool each), and the batch for the team closure.

### `SessionContext` — collapse the two item2/3 fields into `tool_config`

Replace the separate `edit_project_check` + `read_file_max_chars` fields (added in item2.4 / item3.3d) with a single:

```rust
pub tool_config: core_tools::ToolConfig,
```

This is the one translation point where config crosses from the async world (config lock) into the sync construction world.

### Call-site changes

- **`builtin_descriptors.rs`** (boot-time descriptor registration): the per-session `builder` closures for the 9 core tools call `core_tools::<fn>(&ctx.workspace, &ctx.tool_config)` (the 7 plain tools call the no-config fns; edit/read pass `&ctx.tool_config`). The boot-time **probe** (`Tool::new(probe_ws)` for name/description/schema) is unchanged — config does not affect name/desc/schema.
- **`registry_build.rs`** `build_tool_registry` (async): the existing "read config → resolve" block produces a `ToolConfig`, placed into `SessionContext.tool_config`. The async-only tools (browser/skill/memu/plugins) stay inline after `build_session_registry`.
- **`tauri_commands.rs`** `delegate_factory` closure: the `*_for_factory` config locals collapse into a `let tool_cfg = ToolConfig { … }` (built from the values already pre-extracted in the async block before the closure). The 8 hand-written `tool_reg.register(builtin::…)` core-tool lines become one `core_tools::register_core_tools(&mut tool_reg, &workspace, &tool_cfg);`. Then it adds AskUser, ExitPlanMode, plugins as today.

## Data flow

```
async world (reads memubot_config lock)
  ├─ main session  build_tool_registry ──┐
  └─ team prep block ────────────────────┤→ each produces a ToolConfig
                                          ↓
        sync  core_tools::register_core_tools(reg, ws, &cfg)   ← 9 core: single construction + config wiring
                                          ↓
   main: + browser/skill/memu/plugins (inline, async)
   team: + AskUser/ExitPlanMode/plugins
```

Config is *resolved* in two places (the two async paths read the lock at different times — unchanged), but *construction + wiring* happens in exactly one place (`core_tools`).

## Invariants & boundaries

- `core_tools` is pure sync, depends only on `&Path` + `&ToolConfig`, never `&AppState`. This is what lets the sync team closure reuse it without crossing #610's async boundary.
- Descriptor **probe** path (boot) unchanged.
- **Only behavior change** (approved): team sub-agents gain `HttpRequest` (to match the full 9-core set; harmless — same class as the WebFetch they already have). Otherwise zero behavior change — same workspace, same config defaults (project-check off, cap = `MAX_READ_CHARS`).

## Error handling

No new failure modes. Construction is infallible (`Tool::new` + builder methods don't fail). The config-resolution async reads already exist and are unchanged.

## Testing

1. **`core_tools` unit tests:** `register_core_tools` registers exactly the 9 named core tools; `edit_tool`/`read_file_tool` apply project-check / custom cap when `ToolConfig` enables them, and produce default behavior otherwise.
2. **Parity tests:** the team-closure-built registry contains exactly {9 core + AskUser + ExitPlanMode + plugins}; the main-session registry contains {9 core + browser/skill/memu + plugins}.
3. **Regression:** existing item2/3 tests (edit_verify, file, api) stay green — config still reaches the main session via `SessionContext.tool_config`.
4. `cargo build` + clippy clean on touched files; `cargo test --lib agent` net green (only the 2 known pre-existing failures: `shell::test_daemon_mode_approval_unchanged`, `skill_marketplace::truncate_for_error_long`).

## Scope / files

| File | Change |
|---|---|
| `agent/tools/core_tools.rs` | **new** — `ToolConfig` + 9 per-tool constructors + `register_core_tools` + tests (~120 + tests) |
| `agent/tools/mod.rs` | declare `core_tools` |
| `agent/api/session_context.rs` | replace `edit_project_check`+`read_file_max_chars` fields with `tool_config: ToolConfig` |
| `agent/tools/registry_build.rs` | produce a `ToolConfig`; put it in the ctx |
| `agent/tools/builtin_descriptors.rs` | 9 core builder closures call `core_tools::*` |
| `tauri_commands.rs` | team closure: 8 inline registers → `register_core_tools` one-liner; config locals → one `ToolConfig` |

Out of scope: migrating browser/skill/memu/plugins to descriptors (async-descriptor support — the rejected "full P3-2.5" option); decoupling `SessionContext` from `&AppState` (the rejected "true one-handle" option). The `builtin_descriptors.rs` "P3-2.5 follow-up" comment is updated to reflect that core-tool construction is now consolidated and the async tools remain intentionally inline.

## Risk

Medium-low. Pure refactor; default behavior preserved (one approved additive change: team gains HttpRequest). One branch, bisectable commits. Highest-risk touch is `SessionContext` (a #610 shared type) — mitigated by it having a single constructor (registry_build.rs) and the parity/regression tests.
